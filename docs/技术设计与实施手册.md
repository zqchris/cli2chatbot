# cli2chatbot 技术设计与实施手册

## 1. 文档目标

这份文档不是产品介绍，而是为了提高后续继续开发时的命中率。

核心目标：

- 把技术边界讲清楚，避免后续一边写一边重想架构
- 把运行模型固定下来，避免在 `PTY / tmux / 官方 SDK / CLI 包装` 之间来回摇摆
- 把“当前实现是什么”和“下一阶段应该怎么演进”区分开
- 让下一次继续动代码时，优先补最关键的短板，而不是再发散

## 2. 当前架构定位

当前仓库实现的是一个 **本地 bridge daemon + Telegram 私聊控制 + 本地 Web 监控面板** 的 v1 基础版。

当前模型可以概括为：

1. Telegram 私聊发送命令
2. 本地 daemon 轮询 Telegram
3. daemon 选择当前实例
4. 对 `codex` 或 `claude` 启动一次 PTY 子任务
5. 读取流式输出并归一化
6. 通过 Telegram `editMessageText` 持续更新一条消息
7. 同步把状态和输出预览写入本地状态文件与 Web 面板

这意味着当前项目本质上是：

- 一个 **任务桥接器**
- 不是深度会话 runtime
- 也不是官方 SDK 驱动的原生远程控制器

## 3. 关键技术决策

### 3.1 为什么先用 Node.js + TypeScript

当前这个项目最难的不是算法，而是工程拼装：

- Telegram Bot API
- PTY 子进程
- 守护进程控制
- 本地 HTTP API
- 本地 Web 面板
- Codex / Claude 双 runtime 兼容

Node.js 在这些点上的组合成本最低，而且可以直接借鉴 `openclaw` 一类项目的工程组织方式。

当前结论：

- 第一阶段继续坚持 `Node.js + TypeScript`
- 不要为了“更底层”过早切 Go/Rust
- 不要为了“写起来快”临时切 Python 再重构

### 3.2 为什么当前先走 PTY 子任务模式

当前实现选择的是：

- 每个实例是“逻辑实例”
- `/ask` 时用 PTY 启动一条真实 CLI 子任务
- 通过 bridge 自己维护“当前实例”的逻辑状态

这是为了先把最重要的闭环跑通：

- 能发任务
- 能看流式输出
- 能 stop/reset/kill
- 能做本地状态管理

这条路线的优点：

- 实现简单
- 容易联调
- 不依赖某个官方 CLI 的私有协议
- 对 Codex / Claude 都适用

这条路线的缺点：

- 深度会话连续性有限
- `reset` 只能做 bridge 层逻辑重置
- 无法天然“接管一个已经在终端里跑着的会话”

当前结论：

- **短期继续沿用 PTY 子任务模式**
- 不要在还没稳定前就切到“全面 tmux 接管”
- 但代码结构要为第二阶段的“附着式 session backend”预留接口

### 3.3 当前推荐的 runtime 抽象

后续 runtime 层建议固定成两层：

1. `RuntimeAdapter`
   - 只关心如何启动、停止、解析输出
   - 不关心 Telegram、不关心 Web
2. `InstanceSupervisor`
   - 管理实例生命周期
   - 负责选中实例、任务状态、恢复、kill、reset

推荐接口：

```ts
type RuntimeAdapter = {
  startTask(input): RuntimeTaskHandle
  supportsAttach: boolean
  supportsNativeReset: boolean
}
```

后续如果要接入：

- tmux backend
- agent SDK backend
- AgentAPI backend

都应该挂在 `RuntimeAdapter` 后面，而不是直接把逻辑写进 Telegram 处理函数里。

## 4. 当前实现与目标实现的差距

### 4.1 当前已经稳定的部分

- CLI 命令骨架
- 本地配置写入
- 状态持久化
- Telegram 轮询
- 白名单校验
- 实例创建
- PTY 启动子任务
- Telegram 流式编辑
- 本地 Web 状态展示
- daemon 感知 CLI 命令

### 4.2 当前最需要补强的部分

#### A. 真实 runtime 输出格式联调

当前的输出归一化器是“足够工作”的版本，但还没有拿真实复杂任务充分验证。

最优先要验证：

- `codex exec --json` 的真实事件 shape
- `claude --print --output-format=stream-json --include-partial-messages` 的真实事件 shape
- 工具调用事件是否需要和文本事件区分展示
- 出错时 stderr 和 stdout 是否需要分别处理

#### B. reset 语义太弱

当前 `reset` 只是：

- 清空 bridge transcript
- 清空当前 task 绑定

它不是：

- runtime 原生会话 reset
- CLI 原生上下文清理

所以当前文档必须把这点写死，避免误以为已具备“真正会话重置”。

#### C. daemon 生命周期不够完整

后续要补：

- daemon PID 文件
- 启动互斥
- orphan reaper
- idle timeout cleanup
- 进程异常退出后状态回收

#### D. 本地面板仍是只读监控

当前 Web 面板适合看状态，不适合操作。

下一步最合理的增强不是“大改前端”，而是先补最小操作按钮：

- 创建实例
- use 当前实例
- stop
- reset
- kill
- 查看 logs

## 5. 推荐演进路线

### 阶段 A：先把当前 v1 打磨可用

目标：把“能跑”变成“你真的能每天用”

必须做的事：

- 真实 Telegram 联调
- 真实 Codex / Claude 输出格式校准
- 修正 stop/reset/kill 的边界行为
- 增加 daemon 启动互斥
- 增加 orphan 恢复
- 增加 Web 面板基础操作按钮

完成标准：

- 你可以用 Telegram 发真实任务
- 能持续看到流式输出
- 任务结束后状态正确
- 卡死时能远程 kill
- daemon 重启后不会把状态搞乱

### 阶段 B：引入“附着式 session backend”

目标：解决“继续已有会话”和“更强上下文保持”

可选方向：

- tmux backend
- Claude 官方 Agent SDK backend
- 类似 AgentAPI 的统一 HTTP 控制层

建议顺序：

1. 先做 tmux backend 试验版
2. 把它藏在 `RuntimeAdapter` 后面
3. 不要一开始替换掉当前 PTY 方案

原因：

- 当前 PTY 模式已经足够支撑 v1
- tmux 模式适合“继续已有会话”，但复杂度明显更高
- 应该把它做成第二种 backend，而不是重写全部控制面

### 阶段 C：更强控制面

目标：把项目从“能用的小桥”升级成“稳定控制台”

再考虑：

- Web 可交互操作
- 多运行模式切换
- 多项目工作区
- 更细粒度权限控制
- 任务历史和日志检索

## 6. 推荐的联调顺序

后续继续开发时，联调顺序建议固定如下：

1. `init`
   - 配置 bot token、allowed user、codex/claude 路径
2. `doctor`
   - 确认 Telegram、CLI、存储、PTY、端口全部通过
3. `serve`
   - 启动 daemon 和本地面板
4. Telegram 测试基础命令
   - `/help`
   - `/status`
   - `/start_codex`
   - `/instances`
5. 发送一条最小 `/ask`
   - 先测短输出
6. 发送一条长一点的 `/ask`
   - 验证流式编辑
7. 在运行中执行 `/stop`
8. 再执行 `/reset`
9. 构造长时间或卡住任务，再执行 `/kill`
10. 重启 daemon，验证状态恢复

不要一上来就测复杂任务。  
先确认“实例管理”和“输出回传”是稳的，再测真实 coding workflow。

## 7. 下一次继续写代码时的优先级

下一轮最值得先做的事情，按顺序建议：

1. 真实 runtime 输出解析校准
2. daemon 启动互斥和恢复
3. Web 面板可操作按钮
4. 更完整的 logs / task history
5. tmux backend 预研

不要优先做的事情：

1. 多用户
2. 群聊支持
3. 公网 dashboard
4. 大而全权限系统
5. 过早切换到另一门语言重写

## 8. 一句话结论

这个项目当前最合理的路线不是“继续凭感觉扩展”，而是：

**先把当前 PTY + daemon + Telegram 流式桥接做稳，再把 tmux / SDK / AgentAPI 一类能力作为第二阶段 backend 能力引入。**
