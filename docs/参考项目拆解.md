# cli2chatbot 参考项目拆解

这份文档只回答一个问题：

**哪些项目值得借鉴，具体借什么，不该借什么。**

## 1. 参考项目结论

当前最值得参考的不是某一个项目，而是三类思路的组合：

- Telegram 远控桥接
- 会话接管/继续
- 通用 coding agent 控制层

## 2. 项目拆解

### A. RichardAtCT / claude-code-telegram

链接：

- [GitHub](https://github.com/RichardAtCT/claude-code-telegram)

它在 README 里明确主打：

- Telegram 远程访问 Claude Code
- 会话保持
- 以自然语言交互，而不是要求用户记很多终端命令

值得借鉴：

- Telegram 私聊作为主入口
- allowlist / approved directory 思路
- “让用户像聊天一样驱动 CLI”的交互方向
- 安装和配置向导化

不该直接照搬：

- 它主要面向 Claude，不是双 runtime
- 目录授权模型更适合单 runtime + 项目访问控制，不一定适合你现在的“实例池 + 本地 daemon”结构

对 `cli2chatbot` 的启发：

- Telegram 不应该只是发 shell 命令，而应该是一个“任务控制面”
- README 和配置流程要尽量像工具产品，而不是实验脚本

### B. JessyTsui / Claude-Code-Remote

链接：

- [GitHub](https://github.com/JessyTsui/Claude-Code-Remote)

它最有价值的点不是 Telegram 本身，而是：

- 多渠道通知
- PTY mode 和 tmux mode 双模式
- 用 Claude hooks 做状态通知

值得借鉴：

- “默认 PTY，本地直接可用；tmux 作为增强模式”的分层思路
- 通过 hook 通知任务完成 / 等待用户回复
- 引导式 setup，而不是纯手写 `.env`

不该直接照搬：

- 多渠道先不要学，v1 会把复杂度拉高
- hooks 方案在 Codex / Claude 双 runtime 下要重新抽象，不能直接硬塞

对 `cli2chatbot` 的启发：

- 当前 PTY 路线是对的
- tmux 应该作为第二种 backend，而不是现在立刻替换

### C. coleam00 / remote-agentic-coding-system

链接：

- [GitHub](https://github.com/coleam00/remote-agentic-coding-system)

它明确支持：

- Claude Code
- Codex
- Telegram / GitHub 等多平台
- 持久 session

值得借鉴：

- 双 runtime 抽象
- 消息渠道和 runtime 的分层
- “实时流式 or 批量回传”的能力开关
- Docker / remote deployment 友好思路

不该直接照搬：

- 它的系统更大、更偏平台化
- 现在的 `cli2chatbot` 还不需要把 GitHub、数据库、通用命令系统一次全搬进来

对 `cli2chatbot` 的启发：

- 架构上应该继续坚持“渠道层”和“runtime 层”分离
- 后续如果要扩 Slack/GitHub，不该重写 Telegram 那一套，而是加 channel adapter

### D. coder / agentapi

链接：

- [GitHub](https://github.com/coder/agentapi)

它的价值在于一个非常明确的方向：

- 把 Claude Code、Codex 等 coding agent 封装成统一 HTTP API

值得借鉴：

- 统一 agent 控制层
- 屏蔽不同 CLI 的协议差异
- 让上层调用方只关心通用接口

不该直接照搬：

- 现在不需要把 `cli2chatbot` 直接做成一个通用 agent 平台
- 也不需要过早追求“支持所有 coding agent”

对 `cli2chatbot` 的启发：

- 当前的 `RuntimeAdapter` 方向必须保留
- 以后 Telegram / Web / CLI 这些上层，不应该知道 Codex 和 Claude 的细节差异

### E. rclaude

链接：

- [PyPI](https://pypi.org/project/rclaude/)

它最值得关注的是：

- Session Teleportation
- Telegram 和终端之间切换会话
- 按需启动服务器

值得借鉴：

- “会话接管”是独立能力，不是普通 `/ask` 的扩展
- 终端和手机之间切换同一会话，是强需求
- 自动启动 / 闲置关闭这类服务生命周期设计值得后面参考

不该直接照搬：

- 它是 Claude-only
- 它的核心卖点是 teleportation，不是双 runtime bridge

对 `cli2chatbot` 的启发：

- 第二阶段的重点功能应该是“attach existing session”
- 但不应该影响当前 PTY v1 主线

### F. yottoCode

链接：

- [官网](https://yottocode.com/)

它最值得学的是产品形态，不是开源实现：

- Telegram 里直接做权限按钮
- 会话恢复
- 模型切换
- MCP 集成

值得借鉴：

- Telegram 不只是文本输入框，还可以有更强交互控件
- 以后权限审批可以做成按钮式

不该直接照搬：

- 当前项目不需要一开始就追求“像 app 一样完整”
- 也不该在还没稳定前就把 MCP、语音、图片、权限键盘全加进来

对 `cli2chatbot` 的启发：

- 后续 UI/交互增强方向是对的
- 但必须等基础桥接稳定以后再做

## 3. 最终借鉴策略

### 应该立刻采用的

- `claude-code-telegram` 的 Telegram 任务交互方向
- `Claude-Code-Remote` 的“PTY 为主、tmux 为辅”思路
- `remote-agentic-coding-system` 的双 runtime 分层思路
- `agentapi` 的统一 adapter 思路

### 应该暂缓的

- 多渠道接入
- 数据库和复杂平台化
- 公网控制面
- 深权限体系
- MCP / 多模态 / 富交互 Telegram UI

## 4. 对当前项目的结论

`cli2chatbot` 当前最合理的路线不是继续凭直觉叠功能，而是：

1. 保留现在的 PTY + daemon + Telegram 流式桥接主线
2. 把 runtime 抽象继续做干净
3. 下一阶段再引入 tmux / attach 模式
4. 更长远才考虑做成类似 AgentAPI 那样的统一控制层

## 5. 一句话总结

当前最该借鉴的不是“某个项目的全部代码”，而是：

**借 `claude-code-telegram` 的交互，借 `Claude-Code-Remote` 的运行模式，借 `remote-agentic-coding-system` 的双 runtime 分层，借 `agentapi` 的统一控制接口思路。**
